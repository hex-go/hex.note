你是一个程序员，你维护了一个 商品服务，它背后直连 mysql 数据库。  
假设商品服务需要对外提供 **每秒 1w 次查询**，但背后的 mysql 却只能提供每秒 5k 次查询，那 mysql 根本顶不住！分分钟会被压垮。

这类大流量查询场景非常常见，比如双十一秒杀和春节抢车票。  
那么问题就来了，有没有办法在 mysql 不被压垮的同时，让商品服务支持每秒 1w 次查询 ？  

## 1. 本地缓存 

查询内存的速度比查询磁盘要快， mysql 数据主要存放在磁盘里。 

本地缓存方案，就是 查询优先查找服务内部缓存字典，没有结果再查询数据库，再将数据库返回更新放到服务内部缓存。便于后续查询从缓存里命中。将查询请求干到 10k qps 是很轻松的事情。

## 2. 远程缓存

服务实例多时，本地缓存方案，会带来内存浪费。将这部分**字典内存**抽出来，单独做成一个服务。就是**远程缓存服务**。

![图片](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/2/23b97fa2f0630e06/7be51559188c78f5ed68ef4bab3e2be3e92ecdca.jpg "null")

> 如何解决并发带来的data race? 
> 
> 对外不管有多少有个网络连接，收到读写命令后，都统一塞到**一个线程**上，在一个线程上对字典进行读写，规避了 **并发问题** 和 **线程切换开销**

![远程缓存服务 - 简陋](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/2/23b97fa2f0630e06/f73feb604cfc7e312e34c97e9fc206af216fbbea.jpg "null")

这个远程缓存服务足以满足大部分场景，但它属实过于简陋，我们来看下怎么优化它。

### 2.1 多种数据类型支持
> 如何满足不同的业务场景需求？
> 
> 扩展value
> - 字符串
> - List FIFO队列
> - Set  去重
> - ZSet 排行榜

![优化 - 多数据类型](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/2/23b97fa2f0630e06/81893d6d7a7bde47973e44870d06eed8b4de1ee8.jpg "null")

### 2.2 内存过期策略
> 如何限制内存增长速度，减轻服务负担？
> 
> 数据过期时间：**用户通过 expire 命令的形式来指定哪些数据多久过期**，数据过期则删除。

### 2.3 缓存淘汰
> 不设置过期策略的数据，如何减轻服务负担？
> 
> 缓存淘汰：在内存接近上限的时候，根据**一些策略**删除掉一些内存。
> - 比如可以将最近最少使用的内存删掉，也就是所谓的 **LRU**，这样不仅解决了内存过大的问题，还让 redis 里的数据**全是热点数据**。

![缓存淘汰 - LRU](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/2/23b97fa2f0630e06/904ce0625cde79a6f4daa0b9cc43c1312920f215.jpg "null")

### 2.4 持久化
> 缓存服务的数据都在内存中，如何解决服务重启，数据丢失问题？
> 
> **RDB**（`Redis Database Backup`）：缓存服务增加**异步线程**，定期将全量内存数据 **持久化到.rdb磁盘文件**里（将内存数据 → 生成快照 → 保存到文件）。

![Redis Database Backup](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/2/23b97fa2f0630e06/05d3a4ffeeac15a1359ca2b7b1baf4a8141dbc5a.jpg "null")

> RDB间隔几分钟，生成快照期间的数据如何持久化？  
> 
> **AOF**（`Append Only File`）：在每次写入数据时，将数据记录到 `文件缓存` 中，并 **每秒** 将 `文件缓存` 刷入 `磁盘`。服务启动时跟着文件重新执行一遍就能将大部分数据还原，最大程度保证了数据持久化。

![图片](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/2/23b97fa2f0630e06/35f554fe3ce4eae9eb9e133cb449b76dcdf43bcb.jpg "null")

> 如何解决AOF文件过大？
> 
> **定期`重写压缩`**，比如 a 被依次赋值 a=1，a=2，最终只保留 a=2。

### 2.5 简化网络协议

> 缓存服务对外提供的连接协议？
> 
> **直接基于 TCP**，且简化。比如只要通过 TCP 传入 `SET key value`，就能完成写入。传入`GET key` 就能获得对应的 value。非常简洁。  

> 传输协议的解析，需要业务侧实现吗？  
>
> **客户端SDK**，负责协议解析，读写 Redis 服务器里的各种内存数据。

![redis-cli](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/2/23b97fa2f0630e06/ec9ce6a9bf2789d9c6f7fff763278dff2571be99.jpg "null")

## 3. Redis 是什么？
**Redis**，全称 **Remote Dictionary Server**，远程字典服务。
- **高性能**
- **支持多种数据类型**
- **缓存淘汰策略**
- 并提供一定**持久化**

![Redis架构](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/2/23b97fa2f0630e06/a1577acfb216a5dd5a25b47cd79d8665a59303ae.jpg "null")

 
通过扩展插件，还能实现其他高阶能力：  
- **RedisJSON** 支持 **复杂的JSON查询和更新**，≈ 内存版本的 **MongoDB**。  
- **RediSearch** 支持 **全文搜索**，≈ 内存丐版的 **ElasticSearch**。  
- **RedisGraph** 支持 **图数据库功能**，类似 **Neo4j**
- **RedisTimeSeries** 处理时间序列数据，也就是内存版 **InfluxDB**。
大有一种要在内存里将所有中间件都实现一遍的味道。

## 4. 总结

- redis 本质上就是个远程字典服务，所有读写命令等核心逻辑，都在 **单线程** 上完成。不存在并发问题和线程切换开销。
- redis 支持 `多种数据类型`、`内存过期策略` 和 `多种缓存失效策略`，通过 TCP 对外提供了一套非常简单的传输协议。
- redis 加入了最大程度的持久化保证。将数据持久化为 `rdb` 和 `AOF`，确保服务重启后数据不丢。
- redis 支持多种扩展，玩法非常多，比如 RediSearch 和 RedisJSON。
    

最后遗留一个问题，redis 到目前为止就是个**单机**服务，**高性能**是有了，但**高可用**和**可扩展性**是一点没看到。  
这就需要聊聊 **主从** 、**哨兵**和**集群**模式了。
