## 1. 什么是 mysql

mysql是数据和应用的中间层。

![mysql架构总览图](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/878a00d7421134032683e70f350e25167c83e831.jpg "null")

- mysql 分为 `server 层` 和 `存储引擎层`。存储引擎层可更换，既可以是 `myisam`，也可以是 `innodb`。当前用 innodb 更多。
    
- innodb 分为 `内存` 和 `磁盘` 两部分，一部分是 **内存** 里的: `自适应哈希索引`，`buffer bool`，`redo log buffer`。另一部分是 **磁盘** 里存放 行数据和索引的 `.ibd 文件`, 以及 `undo log`, `redo log` 等文件。
    
- `mysql server 层`会通过 `binlog` 记录数据库变更操作，binlog 可以用于 **数据恢复** ，也可以用于 **主从同步** 等场景。

*** 

## 2. InnoDb引擎实现

mysql 将数据组织成 **数据表**（.ibd 后缀的文件）。


### 2.1 数据页（单个数据表，大文件读写慢）

读写单个大文件很慢，mysql就将**数据表拆分成多个数据页**，每页大小 **16KB**。

![图片](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/5ac594bbf32c46fb50f45efcddcc18b22f3a5e53.jpg "null")


### 2.2 索引（数据从哪个数据页找）

> 但数据页那么多，查某条数据时，怎么知道要读哪些数据页？

每行数据增加**行号**，每个数据页加入**页号**，这个 **行号就是主键**。

**B+树索引：** 按主键大小排序，将每个数据页 **最小的行号** 和 **所在页的页号** 提出来，放入到一个 **新生成的数据页** 中，并且给数据页加入 **层级** 的概念（叶子节点只存储数据，其他节点只存储索引，**扇出** = 索引节点有多少子节点，取决于行大小（键值+指针6字节）、页大小，SUM（child）= 16*1024 /(8+6) ≈ 1170）。树扇出大，层数就小，树就矮。

**优点**：
- 扇出大（子节点个数，BigInt约 1170），则层数小（通常3~4层），层数 == 需要查询的数据页个数 == 查找所需的磁盘IO次数，因此对比红黑树（二叉平衡树）的扇出2，性能强了很多；
- 非叶子节点只存储索引，比起B树，能存储更多索引，提高扇出，降低磁盘IO查询次数；
- 叶子节点 + 双向链表串联，范围查询只需要定位起点，之后通过链表获取后续数据；

![图片](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/5c48610d50722b3ec8167edb42d9bc3d1e646fb0.jpg "null")


上面的过程是针对主键的索引，也就是**主键索引**。

![图片](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/f958d913e2b9b2dfb19482f50ca45d7c080b3ad2.jpg "null")


按同样的思路，为数据表的 **其他列** 建立索引，就是**辅助索引**（比如用户表的名称字段，这样就能快速查找到名字为 xx 的用户有哪些）。

![辅助索引](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/de2cc35e97ecbeea5dc76b359de53fe9766bd229.jpg "null")


### 2.3 Buffer Pool（解决磁盘性能瓶颈）

> 数据表、页还是在**磁盘**上。每次都读磁盘太慢了。有办法提升性能吗？

有！在 **磁盘** 和 **应用** 之间，加一层进程内 **缓存**，缓存里就是 16KB 数据页和索引页, 就是 **Buffer Pool**。

![图片](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/eafb443f52590bdec5fcc4e45409a9979e17cd95.jpg "null")

#### 2.3.1 buffer pool是什么

读数据的时候优先读 Buffer Pool，有数据就返回，没数据才去磁盘里读取，减少了读磁盘的次数，大大提升了性能。

> 文件读取，本就会将文件数据加载到**操作系统的文件缓存**中，都是缓存，为什么要设计 Buffer Pool？
>
> 因为，进程自己维护的 Buffer Pool ，可以定制更多 **缓存策略** ，还能实现 **加锁** 等数据表高级特性。

因此，mysql利用 Buffer Pool 通过"**直接 I/O**" 模式, **绕过**操作系统的缓存机制，**直接**从磁盘读写数据。

![BufferPool 直接I/O模式](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/4e2fdffc12356da2ab3ac37ba6475705a7cb745c.jpg "null")


### 2.4 自适应 hash 索引（优化 读性能）

> 利用 buffer pool 解决了磁盘性能瓶颈，但查找 B+树，查询复杂度 **O(logn)**。能更快吗？

能！利用查询复杂度为 O(1) 的 hash 表进行优化。

记录每个数据页的查询频率，对于热点数据页，以查询的**值**为 **key**，**数据页地址**为 **value**，构建 hash 表。

比如，name为 'xiaobai' 的数据页被频繁查询，那 **key** 就是 xiaobai，**value** 就是包含 xiaobai 记录的数据页的地址。

![哈希的key和value](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/ae045977b548e6f0ca9b8bd79ca84a6097ebcf08.jpg "null")

这个 hash 表，就是所谓的**自适应哈希索引**，**Adaptive Hash Index**。

![自适应哈希索引](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/0a5b41d1bed450cfa32717b339557cbb2632d072.jpg "null")


### 2.5 Change Buffer(优化 写性能)

> **自适应 hash 索引**，提高了 **读性能** 。**写性能** 也能优化吗？

问题：当数据表，存在 主键索引 + 辅助索引 ，写操作时，更新完主键索引的**数据页**之后，还需要更新**辅助索引页**。这样读取辅助索引页的磁盘 IO 必然少不了。

定位：Change Buffer 是 InnoDB 引擎用于优化**辅助索引**（非主键索引）**写操作** 的 **缓存结构**。它的设计目标很明确：**减少随机磁盘 I/O，提升写入性能**。

![更新主键索引、辅助索引](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/8ebe10dd15a177dc6efad2dc20220f6319982874.jpg "null")

**Change Buffer**，它其实是 Buffer pool 的一部分。下图对应步骤：
1. 应用 **发起写操作**（触发对索引的修改）；
2. InnoDb检查，写操作需要更新的辅助索引页，是否在buffer Pool中？
   Y：直接进入该页，不走 Change Buffer;
   N: Next；
3. **收集Change、记入内存**。将 写操作 记录到 ChangeBuffer 内存中（不是数据本身，而是将 **对数据的操作** 作为条目，记入 ChangeBuffer 内存中，同时追加 RedoLog中，保证崩溃恢复）；
4. **后续合并、闲时磁盘**。
   时机：**当该索引页被其他查询需要，读入BufferPool时**，先合并ChangeBuffer中的写操作，再供查询；
   动作：将ChangeBuffer中缓存的写操作，真正应用到索引页上，更新该页内容；
   最终：合并后的索引页（变为脏页），在check point或空闲时，写回磁盘。

![Change Buffer的更新流程](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/f83461cb82d3ddc8bba53688ca94de05106f77e9.jpg "null")


### 2.6 Undo Log（先记 旧值，再写 新值）

**事务**：多行数据变更的 **原子性**，要么**同时**更新成功，要么**同时**更新失败。

![事务](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/8eb00e963483ade93dbf7b473c0e6ec261d30eaa.jpg "null")

为了实现事务，就需要知道写数据时每行数据**原来长啥样**，方便对更新后的数据行，进行**回滚**，因此就有了 **Undo Log**。

![undo log回滚](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/959be56cfeca84dd13e7fd1087c2d13fcdc6fd96.jpg "null")


1. 应用发起请求
2. 关键：真正修改 BufferPool 里的数据之前，先把即将被覆盖的 `旧值` 提取出来，存入 `Undo Log` 区域，保证了回滚操作。
3. 持久化保存旧值。为保证事务在服务崩溃后还能回滚，内存 `Undo Log` 区域会被批量写入 `undo log 文件` 中（时机：数据页写入磁盘之前完成，即 WAL Write-Ahead-Logging 预写日志）。

![undo log的写入流程](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/e86cb4fdaf76607194046afe7cb70848cc60f317.jpg "null")


### 2.7 Redo Log

上面内容，主要围绕 `buffer pool` 展开，它们本质上都是内存。

> 如果内存数据只写了一半到磁盘中，数据库进程就崩了，那**一个事务**里的多行数据就没能做到"同时更新成功"。怎么办呢？

将事务中更新数据行的操作都写入到 `redo log buffer` 内存中，然后在 **事务提交** 的时候进行 `redo log` 刷磁盘，将数据固化到 `redo log 文件` 中。

数据库**进程崩溃重启**后，就能通过 `redo log` 找到历史操作记录，**重做**数据。保证了事务里的多行数据变更，要么都成功，要么都失败。

![redo log的写入流程](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/294161550c73f7bab126423b2d50982589161cd3.jpg "null")


> 与其更新 redo log file 文件，直接将 buffer pool 的数据写入到磁盘不香吗？

![为什么不直接写磁盘数据页](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/767e2521d0e3f46d75e6142a702433053fae3389.jpg "null")

**Write-Ahead Logging (WAL)**: 存储系统在 `写数据时` 都会搞个 `日志` 来 `记录操作`，方便 `服务重启后` 进行 `数据对账` ，确保数据的 `一致性` 和 `完整性`。

**因为日志文件是顺序写，顺序写磁盘性能是随机写的几十倍**。

所以，`redo log file` 是**顺序追加**写入的，`buffer pool` 的内存数据是**随机分散**在`磁盘各处`的，

![顺序写比随机写快很多](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/535f70e0b5d44c1c34f100808af228f5fe36d80a.jpg "null")


> 但问题又来了，redo log buffer 也是内存，buffer pool 也是内存，如果 redo log buffer 里的数据还没来得及写入到 redo log，数据库进程就崩了，那 redo log buffer 里的数据不也丢了吗？

是的，所以 `redo log` 的作用并不是保证所有 `数据` 不丢失，而是确保已提交事务的 `变更` 不会丢失。但因为 redo log 刷盘频率很高，所以丢失数据的概率很低。

**redo log 本质上是 `写入性能` 和 `数据完整性` 折中的产物**。

## 3. Innodb 是什么

**innodb 存储引擎**，分为**内存**和**磁盘**两部分，一部分是内存里的`自适应哈希`，`buffer pool`，以及 `redo log buffer`。另一部分是磁盘里存放行数据和索引的`.ibd文件`, 以及 `undo log`, `redo log` 等文件。并对外提供一系列函数接口。

比如操作数据行的 `write_row()`, `update_row()`，以及操作数据表的 `create()`, `drop()`等等接口。

SQL 语句，最终都会转换成 InnoDB 提供的这些接口函数调用。

![innodb提供的函数接口](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/cb583633ec22cc071ef90d5c8588faeabb50e236.jpg "null")

innodb提供的 **函数接口** ，比如：

- `INSERT 语句` 会调用 `write_row()` 接口来 **插入数据行**。
    
- `UPDATE 语句` 会调用 `update_row()` 接口来 **更新数据行**。
    
- `CREATE TABLE 语句` 会调用 `create()` 接口来 **创建新表**。
    
- `DROP TABLE 语句` 会调用 `drop()` 接口来 **删除表**。
    

> 读写 mysql 用的 sql 语句，是怎么转成存储引擎的函数接口的呢？
>
> `Server层`

## 4. Server层 是什么

**Server 层**，本质上是 **`sql语句` 和 `InnoDb存储引擎` 之间的中间层** 。

![server层](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/57a783481b39475a7d2a6d8916683d7d179506d1.jpg "null")

Server层提供的能力：
- **连接管理模块**，用于管理来自应用的网络连接。

- **分析器**，用于判断 SQL 语句有没有语法错误，比如 select，是不是少打了一个`l`。

- **优化器**，用于**根据一定的规则选择该用什么索引**，生成执行计划。

- **执行器**，根据 执行计划 去调用**Innodb 存储引擎**的接口函数。

![server层做了哪些事情](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/8bdb23f6bab0122a4d957c1b7eb36ffec41da40d.jpg "null")


`Server层` 和 `存储引擎层` 是通过 `接口函数` 进行 **解耦** 的，换句话说就是，只要实现了上面这些接口函数，就能作为存储引擎与 Server 层对接。

![引擎解耦](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/77e95838552cc42e66f2d087bc828912a5ea3139.jpg "null")

比如，mysql 早期用的是 **myisam** 存储引擎，后来才支持的 **innodb**。

![常用存储引擎](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/f7e4be9cbcc13b492c7648c4832339c976c120a8.jpg "null")

### 4.2 binlog 是什么

为了防止数据库表被删除带来的影响， `Server层` 会将历史上 **所有变更操作 记录到磁盘上的日志文件中**，这个日志文件就是 **binlog**。一旦误删表，就可以利用 binlog 来恢复数据。
`binlog`主要作用：
1. 主从复制： 从库拿到 Binlog，模拟重新执行一遍 SQL，就能得到一模一样的数据。
2. 时间点恢复： 比如你全量备份了昨天的数据，但今天下午3点误删了一个表。可以用全量备份 + Binlog（恢复到今天下午3点之前）把数据救回来。
3. 审计：谁在什么时间修改了数据。

> `InnoDb引擎` **redo log** 也能做类似的事情，`Server层` 为什么还要设计 **binlog**？
> 
> 原因有二：
> 1. `redo log` 是**环状**写入的，后面写的内容最终会覆盖前面的内容，也就是不会记录所有历史写操作，而 `binlog` 却会记录**所有**历史变更。
> 2. `binlog` 位于 `Server层`，这样不管底层的存储引擎是什么，都能复用这部分能力。
> 
> 一句话总结：
> - RedoLog 保证了 主库 即使宕机也不丢数据（灾难恢复）。
> - Binlog 保证了 从库 可以复制主库的每一次变更（横向扩展），且允许我们对历史上任意时间点进行回溯救人（误操作恢复）。

![binlog写入流程](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/7/7ce7cf5edb037a15/89127b94c7a331faaa3d98682ae3622e3b0a43bf.jpg "null")

### 4.3 Mysql 主从架构

**mysql 主从同步架构**：为了让Mysql承接更高的读写流量，利用 **`binlog`数据**（因为 binlog 记录了一个 mysql 的所有变更操作），"**复制**"一个新的 mysql`从数据库`，承接 **读流量**，原始节点作为 `主数据库` 承接 **写流量**。`从库` 监听 `主库`binlog，同步数据。

**主从同步机制**：
- **`主数据库`**
  - **`Binlog Dump线程`**：与 `从数据库` `IO线程`建立长连接，相应PULL请求；
- **`从数据库`**
  - **`IO线程`**，与`主数据库`建立TCP长连接，拉取 `binlog`日志 → 写入 `Relay log(中继日志)`；
  - **`SQL线程`**，持续监控 `Relay log`，读取 `Relay log` → 执行SQL → 更新数据；

## 5. 示例：读、写流程

接下来用实际例子将上面提到的内容串起来。

首先不管是查询还是更新操作，客户端都会先跟 `Server层` **建立网络连接**，并将 `sql` 发送到 **`Server 层`**，经过 **`分析器`** 解析 `sql` 语法、**`优化器`** 选择`索引`生成`执行计划`，最终给到 **`执行器`** 调用 `InnoDB` 的函数接口。

- 对于**读操作**。`InnoDB` 会先检查 **`Buffer Pool`** 中是否存在所需的 **`B+树数据页`**，如果 **存在** 则直接返回数据。如果 `Buffer Pool` 中**没有**所需的数据页，则会从**磁盘**中读取相应的 `数据页` 加载到 `Buffer Pool` 中，再返回数据。同时，如果查询的数据是**热点数据**，还会将 `数据页` 加入到 **`自适应哈希索引`** 中，加速后续的查询。
    
- 对于**写操作**，`InnoDB`会先将数据写入 **`Buffer Pool`**，并生成相应的 **`Undo Log` 记录**，以便在**事务回滚**时能够恢复数据的原始状态。接下来，会将写操作记录到 **`Redo Log Buffer`** 中，这些 `redo log` 会**周期性**地写入到**磁盘**的 `Redo Log 文件` 中，保证数据库崩溃，已提交的事务也不会丢失。对于 **`辅助索引`的更新操作**，`InnoDB` 会将这些 `更新` 暂时存储在 **`Change Buffer`** 中，**等到相关的`索引页`被读取到 Buffer Pool 时 再进行实际的更新操作**，从而减少磁盘 I/O，提高写入性能。同时，**所有的变更都会记录到 `Server层` 的 `binlog`** 中，以便进行数据恢复。
