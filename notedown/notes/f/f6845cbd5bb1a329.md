## 1. QPS 是什么

**QPS（Queries Per Second）**,也就是“**每秒查询数**”。比如说服务的用户查询 API 支持 100 QPS，就是指这个接口可以做到每秒查 100 次。

![图片](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/f/f6845cbd5bb1a329/8914e16917f5da5d51a3e4c7b32a6faa9840e5af.png "null")

## 2. 合理的 QPS 范围

带了数据库的服务一般写性能在 5k 以下，读性能一般在 10k 以下，能到 10k 以上的话，那很可能是在数据库前面加了层缓存。如果你的服务还带了个文本算法模型，那使用了 gpu 的情况下 API 一般支持 100~400QPS 左右，如果是个同时支持文本和图片的模型，也就是所谓的多模态模型，那一般在 100QPS 以内。

![图片](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/f/f6845cbd5bb1a329/a29147cbba519083a78f4ab63e830d010ca98cd1.png "null")


## 3. 如何计算 QPS ？

现在了解完 QPS 了，假设我们想要获得某个函数 的 QPS，该怎么做呢？  
这一般分两个情况：

1. **实时性要求较低**的监控场景。
    
2. **实时性要求较高**的服务治理场景。
    

![计算qps的两个场景](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/f/f6845cbd5bb1a329/d3892af0fa7aca26c16147c2f678d47fe1cd5b44.png "null")


### 3.1 监控场景

监控服务 QPS 是最常见的场景，它对实时性要求不高。如果我们想要查看服务的 QPS，可以在服务代码内部接入 `Prometheus` 的代码库，然后在每个需要计算 QPS 的地方，加入类似`Counter.Inc()`这样的代码，意思是函数执行次数加 1。这个过程也就是所谓的**打点**。

当函数执行到打点函数时，Prometheus 代码库内部会计算这个函数的调用次数，将数据写入到 `counter_xx.db` 的文件中，再同步到公司的`时序数据库`中，然后我们可以通过一些监控面板，比如 `grafana`调取时序数据库里的打点数据，在监控面板上通过特殊的表达式，也就是`PromQL`，对某段时间里的打点进行求导计算速率，这样就能看到这个函数的调用 QPS 啦。

![监控场景中获取qps](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/f/f6845cbd5bb1a329/0a6fe96a82e02da0645eb07e468b8a2808bbc90f.png "null")

### 3.2 服务治理场景

跟监控面板查看服务 QPS 不同的是，我们有时候需要以**更高的实时性**获取 QPS。

比如在**服务治理**这一块，我们需要在服务内部加入一些中间层，实时计算服务 api 当前的 QPS，当它大于某个阈值时，可以做一些自定义逻辑，比如是直接拒绝掉一些请求，还是将请求排队等一段时间后再处理等等，也就是所谓的**限流**。  
这样的场景都要求我们实时计算出准确的 QPS，那么接下来就来看下这是怎么实现的？

#### 3.2.1 基本思路

某个函数的执行 QPS == 每秒内这个函数被执行了多少次。

可以参考监控场景的思路，用一个**临时变量 cnt** 记录某个函数的**执行次数**，每执行一次就给变量`+1`，然后计算单位时间内的变化速率。公式就像这样：

```
QPS = (cnt(t) - cnt(t - Δt)) / Δt
```

- `cnt(t)`: 表示在时间 `t` 的请求数
- `Δt`：表示时间间隔

比如，第 9 秒, cnt 是 80，第 10 秒，cnt 是 100，那这一秒内执行了 `(100-80)/(10-9) = 20 次`, 也就是 20QPS。

![QPS怎么计算](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/f/f6845cbd5bb1a329/07a4d42cc62d781fda85106617008f7bde0b78fb.png "null")

#### 3.2.2 引入 bucket数组
> 问题，历史记录会被覆盖，单一变量无法记录连续数据。

到了第 10 秒的时候，如果想知道第 5秒 和第 6 秒的 QPS，光一个变量的话，数据老早被覆盖了，根本不够用。

于是将临时变量 cnt，改成了一个数组，数组里每个元素叫 `bucket`，用来存放(cnt(t) - cnt(t - Δt)) 的值。

![bucket数组](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/f/f6845cbd5bb1a329/f7a8645554027222b36941736a10528b4846f86c.png "null")


#### 3.2.3 调整 bucket 范围粒度

bucket的时间粒度，就是数据采集的粒度，**粒度越细，数据波动越明显，识别的 QPS 越灵敏，基于这个 QPS 做的服务治理能力响应就越快**。原来用 1 个 bucket 存放 1s 内的增量数量，现在就变成要用 5 个 bucket 存放。

![bucket粒度细化](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/f/f6845cbd5bb1a329/b153994bb5756329c512c9fc4b8982747abc2ef1.png "null")

#### 3.2.4 引入环形数组

环状数组，就是**一个用%取模来确定写入位置的定长数组**，避免时间变长，数组长度无限制增长，内存占用过多，导致进程OOM。  

比如数组长度是 5，数组 index 从 0 开始，要写 index=6 的 bucket， 计算 6%5 = 1，那就是写入 index=1 的位置上。

![bucket环形数组](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/f/f6845cbd5bb1a329/793724b3d5acc4489f02555cdb955853cc632cdb.png "null")

#### 3.2.5 加入滑动窗口
要获取1秒内的请求总数，这个时间窗口就是1秒，当前时间对环形数组长度取模，获得end指针位置，再根据bucket粒度往前找到start指针位置，【start:end】就是滑动窗口的范围，将这个范围内的bucket求和，就是1s内的qps

![引入滑动窗口](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/f/f6845cbd5bb1a329/4b2ffbbbfe65d4fa83c69eeadf90a24a1d21bb4a.png "null")

到这里 qps 的计算过程就介绍完了。

#### 如何计算平均耗时

某个函数的**平均耗时**，bucket 再加个用来统计延时的变量 `Latency` ，记录单次函数执行耗时。再通过滑动窗口获得对应的 bucket 数组范围，`函数平均耗时=sum(Latency)/sum(cnt)`。

```
函数的平均耗时 = Latency总和/cnt总和
```

### sentinel-golang

上面的思路，就是阿里开源的sentinel-golang中 QPS 计算组件的实现方式。sentinel-golang 是个著名的服务治理库，它会基于 QPS 和 Latency 等信息提供一系列限流熔断策略。 

基于环形数组、滑动窗口，bucket(cnt|latency)实现的计算 QPS 和 平均Latency 

```
https://github.com/alibaba/sentinel-golang
```

入口：基于 `sliding_window_metric.go` 里的 `GetQPS` 开始，它是实时计算 QPS 的入口函数。这里面会看到很多上面提到的内容细节，其中：
- **滑动窗口**，叫 `LeapArray`
- **bucket环形数组**，叫 `AtomicBucketWrapArray`
- bucket，叫`MetricBucket`，但需要注意的是 MetricBucket 里的 count 并不是一个数字类型，而是一个 map 类型，它将上面提到的 cnt 和 Latency 等都作为一种 key-value 来存放。
