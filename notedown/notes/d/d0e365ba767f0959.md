单点redis存在两个问题：高可用、可扩展(突破单机内存上限)。

> 如何解决高可用问题？
>
> Redis HA模式：增加`从节点`，时刻从`主节点`同步数据。

> 所有节点存的都是**全量数据**，但单机服务器内存总有上限。增加节点无法突破单机容量瓶颈
> 
> **redis 集群模式**：通过数据切分，将数据分散到不同的节点上。

## 1. 介绍
- 从节点：可用性
- 数据切分：扩展性 (哈希槽切分，重定向，数据迁移)

**集群模式**：将 **数据分片到多个`redis节点`** 上，同时 **为每个节点添加从节点**的架构。

![Redis集群](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/d/d0e365ba767f0959/8915ac1f6eb0b98b23206a3bbab3b82a0d7d9aab.png)

## 2.**数据切分**

> 问题：有限的单机内存 无法支持 无限的数据
>
> 既然数据无限，就对数据进行切分，变成多份，将多份`切片数据`放到多个 redis 节点上。通过增加节点数量，扩容集群数据容量。

![切分数据](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/d/d0e365ba767f0959/a89305c15613316d8bee46030315a004053e3549.png)

### 2.1 如何切分？

#### 2.1.1 直接求余

1. **对`redis节点`进行编号**;
2. **再将 `key` 的值** 跟 **`redis`的节点总数** 进行 **求余**，计算数据存放的节点。

![对crc16(key)求余](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/d/d0e365ba767f0959/b4b40cd9fa446bf97b4b9a0446d70f3608261df8.png)

> 求余有个很大的问题：节点的扩展，带来redis节点总数的变化，会导致数据对应节点发生改变，极大概率造成 **所有数据** 都需要 **重新迁移**。
>
> 解决：**引入 `哈希槽` 固定 `求余的值`**（也就是数据落到的节点）

#### 2.1.2 哈希槽

1. `数据key` 和 `redis节点` 之间，加一层 **长度固定（16384）** 的 `数组层`（**数组的每个槽位，就是`哈希槽 slot`**）。
2. 公式改为 **对 `len(数组层)` 求余**，数据将落在槽上。
   ![增加 数组层](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/d/d0e365ba767f0959/083a2922065d1befd77ca914b71c496d6a069452.png)
3. 每个redis节点，分别处理 **部分`哈希槽`**（比如，`Redis 0节点`处理 前1W槽位，`Redis 1节点`处理 后6384槽位）。
![把哈希槽分配给多个节点](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/d/d0e365ba767f0959/b4280561f77ef05998453a7456dbd35de8def4c0.png)

4. `数据的 key` 经过公式分片后会落到某个`哈希槽`上，再由管理这个`槽`的`redis节点`来读写这个数据。
5. **每个节点都维护了 `哈希槽 ↔ 节点 对应关系`**，有了这些信息，任意一个 redis 节点都可以根据key计算数据落点。
6. **增加 redis 节点**，由于哈希槽这一层的存在，`16384` 固定不变，那计算出的 `slot_num` 也不会变化，**数据落的槽位也就不会变**。只需要 **定义 `新增的redis节点` 负责哪部分`哈希槽`，然后 迁移该部分`哈希槽里的数据`** 就好了。

![只需要迁移部分数据](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/d/d0e365ba767f0959/ce839d524bda2ab1499df3e4f8519dc16edd1ae9.png)

引入`哈希槽`切分数据，实现了集群存储数据。但引入了 **该访问哪个节点** 的新问题。
> `哈希槽 ↔ 节点 对应关系` 维护在 每个`redis节点` 上，客户端没有这个信息，读写数据时，怎么知道去哪个节点读取数据？
>
> `客户端` 通过`CLUSTER SLOTS`命令，获取最新的哈希槽分配信息。

> 如果redis节点数量有变化，客户端没有及时更新哈希槽信息，会不会出错？
>
> `客户端` 可以对 **任意`redis节点`读写**，不属于当前节点的key，会返回错误，并告知`客户端`该去哪个节点，这个过程就是 **重定向**。

## 3. **重定向**

接收到数据的 `redis节点` 会根据 `最新维护的哈希槽分配信息`，判断这个 key 属于哪个`redis 节点`，如果恰好属于它自己，则直接完成读写。否则，redis 节点就返回一个错误，并告诉客户端该去哪个 redis 节点，这个过程也叫**重定向**。（客户端对重定向的处理，通常在SDK库中处理，业务代码不关心。）

![重定向](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/d/d0e365ba767f0959/3fa9334a21b84ea27dcc6bb76474490c5dec68f3.png)


## 4. **数据迁移**

> 增删`redis节点`，数据正在发生迁移行为，客户端查某个 redis 节点没数据，如何确定它是 **真没有这个key**，还是**没来得及迁移过来**呢？
>
> redis 会对`哈希槽`维护一个状态：**是否正在迁移数据**。

![维护一个迁移状态](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/d/d0e365ba767f0959/b82650c835e02552f520e35266c117a7a34a8309.png)

如果正在迁移中，redis 节点会返回一个**重定向**指令，告诉客户端接下来该去哪个节点读这个数据，客户端根据重定向指令的提示，重新访问另一个 redis 节点，如果这时候还是没有数据，那就是真没数据。

## 5. 举例：了解工作原理

![Redis集群节点](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/d/d0e365ba767f0959/55994164579fc9642186fc8ac70b8163e7a2a6cd.png)

Redis 集群有 3 个主节点（A、B、C），`哈希槽分配信息`为：

- `节点A` 负责哈希槽 `0~5460`
    
- `节点B` 负责哈希槽 `5461~10922`
    
- `节点C` 负责哈希槽 `10923~16383`
    

### 5.1 **写操作：**

客户端要写入数据：**`key` 是 `user:1001`，`value` 是 `小白 debug`**。

1. 客户端会**随机访问**某一个`redis节点`，比如 `节点A` ，发送 `set key value` 的操作。
   > 注意: 如果客户端有slot信息，则不「随机连」

2. `节点A` 计算`key`对应的哈希槽的值，再根据`哈希槽分配信息`，判断数据所在节点。
   - 比如 `1234`。`节点A`发现 `1234` 在哈希槽 `0~5460`范围内，归它自己管， 所以这个Key会被写入到`节点A`。
   - 如果是`5462`, 属于`节点B`管，`节点A`就会返回一个**重定向指令**，告诉客户端，这个key应该写到 `节点B`，客户端会自动重定向到`节点B`，完成写入。

### 5.2 **读操作：**

客户端要读取的`key`是 `user:1001`。

1. 客户端会**随机访问**某一个`redis节点`，比如`节点A`，发送 `get key` 的操作。
2. `节点A` 计算`key`对应的哈希槽的值，再根据`哈希槽分配信息`，判断数据所在节点。
   - 如果发现 `user:1001` 的哈希槽的值是 `1234`，归`节点A`自己管，那么就会返回对应的值`"小白 debug"`。
   - 但如果计算得到的哈希槽的值是 `5462`，属于`节点B`。那`节点A`会返回一个**重定向**指令，告诉客户端，这个key在`节点B`，客户端会自动重定向到`节点B`，完成数据读取。
