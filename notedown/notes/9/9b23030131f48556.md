随着 Ingress NGINX 于 2026 年 3 月正式退役，企业平台团队正面临一项紧迫的安全与合规任务。继续使用已经退役的控制器，意味着关键基础设施将暴露在无法及时修复的安全风险之下。

对于一位管理着 60 多个复杂 Ingress 资源的基础设施工程师来说，挑战非常明确：找到一个现代化、面向企业场景的替代方案，并且不能投入数月时间进行手动重构。

本文将介绍，如何借助 AI Agent 和 Higress，在短短 30 分钟内完成完整的迁移验证。Higress 是由阿里巴巴发起的云原生、AI 原生 API 网关，近期已加入 CNCF Sandbox。

解决方案：为什么 AI 时代需要 Higress？

Higress 基于业界标准的 Envoy 和 Istio 构建，专为 AI 原生网关场景设计。它不仅解决了传统控制器的不足，也为大语言模型（LLM）提供了专门能力。

AI 原生架构：不同于传统网关，Higress 将 LLM 视为一等公民。它内置了基于 Token 的限流能力，可用于管理模型成本；同时提供缓存能力，用于降低常见 AI Prompt 的响应延迟。

LLM 协议治理：Higress 提供统一协议，用于对接不同的 LLM 服务提供商，使团队可以在单一、安全的入口背后灵活切换模型。

零停机可靠性：借助 Envoy 的 xDS 协议，Higress 可以在毫秒级完成配置更新。这也避免了 “NGINX reload” 问题，对于维持 AI 流式输出和 gRPC 场景中的长连接非常关键。

Model Context Protocol（MCP）：Higress 支持托管 MCP Server，使 AI Agent 能够通过网关安全地访问企业工具和数据。

AI 辅助迁移工作流

为了加速迁移，阿里巴巴工程师使用了配备专门 “Skills” 的 AI Agent。通过这种方式，分析和验证这类繁重工作被交给 AI 处理，而最终的生产执行仍然由人工工程师掌控。

理解当前状态

首先，工程师要求 Agent 审计现有集群。

借助 nginx-to-higress-migration skill，Agent 自动识别出所有 Ingress 资源，并标记出需要转换的 NGINX 特定 annotations。

无风险模拟

为了确保迁移不会破坏生产流量，工程师使用 Agent 基于 Kind（Kubernetes in Docker）创建了一个模拟环境。

在安装 Higress 时，工程师关闭了状态更新能力（global.enableStatus=false），以防止 Higress 更新 Ingress 的 status 字段，从而让它能够与 NGINX 平稳共存。

这样一来，工程师就可以在旧的 NGINX 控制器旁边，并行测试新的路由逻辑。

用 WASM 解决自定义逻辑

对于分析过程中识别出的复杂 NGINX snippets，工程师使用了 higress-wasm-go-plugin skill。

该 skill 让 AI 能够生成高性能 WebAssembly（WASM）插件，用 Higress 沙箱中的 WASM 插件来复刻原有的自定义 Lua 或 NGINX 逻辑。

结果：30 分钟达到合规

通过 Higress 原生的 NGINX 兼容能力，以及 AI 辅助验证，基础设施迁移得以快速完成。

![图片](https://raw.githubusercontent.com/hex-go/hex.note/main/notedown/assets/9/9b23030131f48556/b2a2be057fcd6b99132bdbb5cfc56302be7c62fe.png)

Ingress NGINX 的退役，不只是一次迁移挑战，也提供了一个升级到更具韧性、更适合 AI 场景架构的机会。

迁移到 Higress 后，组织可以获得一个基于 Envoy 和 Istio 的企业级网关，并面向未来的 LLM 集成场景做好准备。

Tianyi Zhang（阿里巴巴 / 阿里云）
